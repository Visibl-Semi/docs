# Doc Gen

Generate comprehensive AI-powered documentation for your RTL modules and projects using an advanced **4-agent agentic architecture**. This sophisticated multi-agent system analyzes your RTL code through multiple specialized AI agents working in coordination to produce professional-grade documentation in Markdown and Google Docs formats.

<Info>
**Advanced AI System**: Each module undergoes deep analysis by 4 specialized AI agents (~30 minutes to 1 hour per module), ensuring comprehensive coverage of functionality, interfaces, timing, performance, and design patterns. This is a long-running process that produces thorough, publication-ready documentation.
</Info>

## Quick Start

Launch the Visibl TUI and use these key commands:

```bash
# Launch Visibl TUI
visibl

# Essential documentation commands:
/docs all                    # Generate docs for configured modules
/docs module1 module2        # Document specific modules
/docs configure-targets      # Set up persistent targets
```

## Key Features

- **4-Agent Agentic Architecture** - Specialized AI agents for comprehensive RTL analysis
- **Deep Technical Analysis** - Covers functionality, interfaces, timing, performance, and design patterns
- **Multiple Output Formats** - Markdown, Google Drive folders, optional DOCX export
- **Background Execution** - Long-running process survives terminal drops with live logging
- **Smart Parsing** - Uses Slang JSON artifacts when available, falls back to regex parsing
- **Publication-Ready Output** - Professional documentation suitable for design reviews and handoffs

## Target Configuration

Edit `docs/settings/targets.txt` with one module name per line:
```
cpu_core
cache_controller
uart_controller
```

Configurations stored in:
- `docs/settings/targets.json` - named profiles
- `docs/settings/targets.txt` - default profile

### Output Structure

Documentation is generated in a structured format:

```
your-project/
├── docs/
│   ├── markdown/                    # Primary documentation
│   │   ├── cpu_core.visibl.md
│   │   ├── cache_controller.visibl.md
│   │   └── memory_arbiter.visibl.md
│   ├── googledocs/                  # DOCX exports
│   │   ├── cpu_core.docx
│   │   └── cache_controller.docx
│   ├── logs/                        # Execution logs
│   │   └── docagent_docs-20240326-143022.log
│   ├── settings/                    # Target configurations
│   │   ├── targets.json            # Profile storage
│   │   └── targets.txt             # Simple text format
│   └── slang/                       # Parser artifacts (when available)
│       ├── modules/
│       ├── hierarchies/
│       └── interfaces/
└── output/                          # Fallback artifacts
└── dependency_graphs/
    └── project_dependency_graph.json
```

## Documentation Content

### Generated Documentation Structure

Each module documentation file (`module_name.visibl.md`) contains:

**Standard Sections:**
- **Block Name** - Module identification and purpose
- **Overview** - High-level functional description
- **High-Level Requirements** - Design specifications and constraints
- **Block Diagram** - Visual representation of module structure
- **Top-Level Block Diagram Description** - Detailed diagram explanation

**Interface Documentation:**
- **IO Table** - Complete port descriptions with signal purposes
- **Parameters and Typedefs** - Configuration parameters and custom types
- **Interfaces** - Protocol and interface specifications
- **Protocols** - Communication protocol details

**Timing and Control:**
- **Clocks** - Clock domain and timing requirements
- **Resets** - Reset behavior and sequences
- **Example Transactions** - Timing diagrams (often includes Wavedrom sketches)

**Performance and Design:**
- **Performance** - Throughput, latency, and resource utilization
- **Arbitration/Fairness/QoS** - Resource management policies
- **Forward Progress Guarantees** - Deadlock prevention and liveness
- **Power** - Power consumption and optimization
- **Physical Design** - Layout and placement considerations

**Advanced Features:**
- **CSRs** - Control and status register descriptions
- **Performance Analysis** - Detailed performance characteristics
- **Debugging** - Debug interfaces and methodologies
- **Reliability** - Error handling and fault tolerance
- **Implementation Details** - Low-level implementation specifics

### Content Generation Features

**Deterministic Content:**
- IO Table and Parameters sections can be generated deterministically from parsed metadata
- Ensures useful baseline documentation even without LLM processing
- Provides structured information extraction from RTL source

**AI-Enhanced Analysis:**
- Functional behavior descriptions generated by AI analysis
- Protocol compliance verification and documentation
- Design pattern recognition and explanation
- Integration guidance and best practices

**Visual Elements:**
- Automatic Table of Contents generation
- Wavedrom timing diagrams for transaction examples
- Block diagrams and architectural overviews
- Cross-referenced module relationships

### Output Formats

**Markdown** (`docs/markdown/`):
- Version control friendly
- GitHub/GitLab integration
- Static site generator compatible

**Google Docs** (`docs/googledocs/`):
- DOCX exports (requires Pandoc installation)
- Ready for collaborative editing


## Integration Workflows

### Git Integration

Include documentation generation in your development workflow:

```bash
# Generate docs before commits
visibl /docs all
git add docs/
git commit -m "Update module documentation"
```

### CI/CD Integration

Automate documentation updates:

```yaml
# .github/workflows/docs.yml
name: Update Documentation
on:
push:
  paths: ['rtl/**/*.sv', 'rtl/**/*.v']
  
jobs:
docs:
  runs-on: ubuntu-latest
  steps:
    - uses: actions/checkout@v3
    - name: Generate Documentation
      run: |
        visibl /docs all
        git add docs/
        git commit -m "Auto-update documentation" || exit 0
        git push
```

### Team Collaboration

Share documentation efficiently:

1. **Markdown for developers** - Version controlled, easy to review
2. **Google Docs for stakeholders** - Collaborative editing, comments
3. **Automated updates** - Keep docs synchronized with code changes

## Best Practices

- Start with top-level modules and critical interfaces
- Use `/docs configure-targets` for persistent project documentation
- Use `/docs module_name` for specific components
- Regenerate docs after significant RTL changes
- Review AI-generated content for accuracy
- Include `docs/markdown/` in version control

## Troubleshooting

### Common Issues

**No documentation generated:**
- Check logs in `docs/logs/docagent_docs-*.log`
- Verify module names match discovered modules
- Ensure write permissions to `docs/` directory

**LLM service connection errors:**
- Start the Visibl LLM service
- Check `VISIBL_BASE_URL` environment variable

**Empty documentation:**
- Ensure RTL files are syntactically correct
- Add detailed comments to improve AI analysis

**DOCX export issues:**
- Install Pandoc and ensure it's in PATH

### Performance Tips

- Use selective targeting for large projects
- Run in background mode (default) for long operations
- Generate documentation in smaller batches if needed

### Debugging

Check logs in `docs/logs/` and use test mode for debugging:
```bash
/docs module_name test
```

## Advanced Features

### Parser Backends
- **Slang JSON Parser** (preferred): Uses `docs/slang/` artifacts for accurate analysis
- **Regex Parser** (fallback): Basic extraction when Slang unavailable

### Execution Options
```bash
/docs all overwrite    # Force regeneration of existing docs
/docs all test         # Generate placeholder content for testing
```

### Background Processing
- Default background execution survives terminal drops
- Live status and logs accessible through TUI
- Automatic process cleanup
